Avoid this rookie mistake when working with database indexing!

In this blog post, I want to reflect a valuable lesson from a mistake that I made when working with PostgreSQL database. I encountered a major hiccup while attempting to create a unique index for a table. My intention was to optimize query performance, but instead, I found myself drowning in redundant data.
Lets dive right in by setting up a table with some initial values. 

CREATE TABLE contact_person (
    id            SERIAL PRIMARY KEY,
    email         TEXT,
    first_name    TEXT,
    last_name     TEXT,
    phone_number  TEXT
);

CREATE UNIQUE INDEX ON contact_person(LOWER(email),
                                      LOWER(phone_number));

INSERT INTO contact_person (email, first_name, last_name, phone_number)
    VALUES ('[email protected]', 'Peter', 'Parker', '034132453');

INSERT INTO CONTACT_PERSON (email, first_name, last_name, phone_number)
    VALUES ('[email protected]', 'Steve', null, null);

Now, the table contact_person will look like this

id email first_name last_name phone_number
1 [email protected] Peter Parker 034132453
2 [email protected] Steve [NULL] [NULL]

So whats the issue here?

My intention is that a row in this table is unique based on a combination of 2 columns (email, phone_number). However, as you can see in the table definition, all these columns are nullable, which means the created index wont work. This is because two NULL values are considered distinct in aUNIQUEconstraint - in accordance with the SQL standard.
We can test that by running this code. 

SELECT NULL = NULL AS equal;

The result is null.
Lets do another test with an upsert in my case. 

INSERT INTO contact_person ( email, first_name, last_name, phone_number)
                    VALUES ('[email protected]', 'Peter', 'Parker', '034132453')
    ON CONFLICT (LOWER(email), LOWER(phone_number))
    DO UPDATE
    SET email = '[email protected]',
        first_name = 'Peter',
        last_name  = 'Smith',
        phone_number = '034132453'
    RETURNING id;

The result is:

id email first_name last_name phone_number
1 [email protected] Peter Smith 034132453
2 [email protected] Steve [NULL] [NULL]

Now I want to update Steves contact with the same operation: 

INSERT INTO contact_person ( email, first_name, last_name, phone_number)
                    VALUES ('[email protected]', 'Steve', 'Roger', null)
    ON CONFLICT (LOWER(email), LOWER(phone_number))
    DO UPDATE
    SET email = '[email protected]',
        first_name = 'Steve',
        last_name  = 'Roger',
        phone_number = null
    RETURNING id;

However, the result is not what I expected:

id email first_name last_name phone_number
1 [email protected] Peter Smith 034132453
2 [email protected] Steve [NULL] [NULL]
4 [email protected] Steve Roger [NULL]

Its supposed to update the row with id = 2.

Solution?

So far we all understand whats causing the issue and creating redundant data to contact_person table. So what is the solution for this?
Thanks to StackExchange, I found 4 possible ways to deal with this issue

For Postgres 15 or newer version

  1. Postgres has a solution traight out of the box 

    CREATE UNIQUE INDEX contact_person_email_phone_number_idx
    ON contact_person (LOWER(email), LOWER(phone_number)) NULLS NOT DISTINCT;

For older versions

  1. Create partial unique indexes 

    CREATE UNIQUE INDEX contact_person_email_idx
                                 ON contact_person(LOWER(email)) WHERE phone_number IS NULL;
CREATE UNIQUE INDEX contact_person_phone_number_idx
                                 ON contact_person(LOWER(phone_number)) WHERE email IS NULL;

  2. Create a unique index with COALESCE 

    CREATE UNIQUE INDEX contact_person_email_phone_number_idx
    ON contact_person (COALESCE(LOWER(email), ''),
                       COALESCE(LOWER(phone_number), ''));

  3. Create aUNIQUEindex or constraint based on a cheap and sufficiently unique hash value of the row (reduced to defining columns). (See this post for more information)

Each of these solutions has pros and cons that are discussed in 2 resources I provide at the end of this blogs. If youre interested in finding the best solution for you, you can look into those 2 URLs for more details.
Since the third solution is good enough for my case, I choose to implement it in my project.

Result

Lets continue to run this code to test my solution 

DELETE FROM contact_person WHERE id = 4;

CREATE UNIQUE INDEX contact_person_email_phone_number_idx
    ON contact_person (COALESCE(LOWER(email), ''),
                       COALESCE(LOWER(phone_number), ''));

The reason I need to delete the row with id = 4 is because the new unique index will consider row with id = 2 and id = 4 are duplicate. If I dont delete one of these rows, I will get following error:

SQL Error [23505]: ERROR: could not create unique index contact_person_email_phone_number_idx Detail: Key (COALESCE(lower(email), ::text), COALESCE(lower(phone_number), ::text))=([email protected], ) is duplicated.

Now if I run this code, I will expect the database to update the record with id = 2 

INSERT INTO contact_person ( email, first_name, last_name, phone_number)
                    VALUES ('[email protected]', 'Steve', 'Roger', null)
    ON CONFLICT (COALESCE(LOWER(email), ''), COALESCE(LOWER(phone_number), ''))
    DO UPDATE
    SET email = '[email protected]',
        first_name = 'Steve',
        last_name  = 'Roger',
        phone_number = null
    RETURNING id;

And voila!

id email first_name last_name phone_number
1 [email protected] Peter Smith 034132453
2 [email protected] Steve Roger [NULL]
